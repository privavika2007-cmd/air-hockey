# Aerohockey — Frontend Design Spec v2

Живой документ. v1 был основан на первом мокапе (6 режимов, тил-кнопки). Эта версия (v2) переписана под актуальное ТЗ Вики + реальные ассеты + реальный backend-код — v1 полностью устарел, см. раздел 8 "Изменения с v1".

**Статус:** v2, на основе полного комплекта ассетов (images/music/sound_effects/stickers/visual_effects) и реального backend-кода (server/).

---

## 1. Роли в команде

- **Вика** — весь фронтенд и клиент: Экран 1 (заставка/меню/выбор режима/счёта/стика/звука) И Экран 2 (игровое поле, рендер матча, сетевой клиент). Это отличается от более ранней версии плана, где экраны были поделены между двумя людьми — сейчас клиент целиком на Вике.
- **Бэкенд-разработчик** — сервер синхронизации (`server/`, уже частично написан, см. раздел 5).

## 2. Стек

Python 3.13 + Pygame 2.6.1 (на клиенте). Бэкенд — FastAPI + WebSocket + uvicorn (уже есть в `server/`).

**Важно:** на Python 3.14 у pygame ломается модуль шрифтов (`pygame.font`) — окружение для клиента создавать строго на Python 3.13 или ниже.

## 3. Экран 1 — Splash / меню / выбор перед стартом

### 3.1 Палитра (ФИНАЛЬНАЯ — заменяет v1)

В v1 был открытый вопрос "кнопки тил или синие" — Вика в новом ТЗ явно подтвердила **синий неон**. Также подтверждено, что иконки (play-треугольник, hamburger, нота) — **жёлтые**, все три, не только play. Это уже так и реализовано в текущем прототипе (`prototype/aerohockey_interactive.py`), там это исторически вышло тил+жёлтый по ошибке — при полировке заливку кнопок надо перекрасить в синий.

| Роль | Значение |
|---|---|
| Фон экрана | `#000000` |
| Заливка круглых кнопок | синий (уточнить точный оттенок у Вики — она не дала hex, ориентир: насыщенный электрик-синий, не голубой) |
| Neon-обводка вокруг кнопок | синий неон (тот же тон, свечение ярче/более desaturated версия заливки) |
| Иконки (play-треугольник, hamburger, note) | жёлтый, **+ drop shadow** для объёма (новое требование — в v1 тени не было) |
| Mode-card / overlay-панель | тёмно-серая полупрозрачная, как в v1 |

### 3.2 Компонент: Circle Button

Круглая кнопка, синяя заливка + синее неоновое свечение по периметру. Иконка внутри — жёлтая, с падающей тенью (реализовать как второй слой иконки, смещённый на несколько px, тёмный/чёрный полупрозрачный, под основной иконкой).

- **Play** (низ по центру, крупная) — жёлтый равносторонний треугольник вершиной вправо.
- **Menu** (право-верх) — три жёлтые горизонтальные полоски (hamburger).
- **Sound** (право-верх, под Menu) — жёлтая восьмая нота.

Все три кликабельны (уже реализовано).

### 3.3 Menu Overlay — выбор тематического режима

**Три режима** (не шесть — см. раздел 8 про несостыковку в тексте ТЗ). Каждый — полупрозрачная серая скруглённая карточка, название центрировано внутри.

| Режим | Название на карточке | Стики | Шайба | Музыка | Доп. оформление карточки |
|---|---|---|---|---|---|
| Космос | **Cosmic Rink** | Земля / Марс | Луна | Star Wars | по бокам карточки — по одному стикеру планеты (см. `stickers/`) |
| Неон (default) | **Neon Velocity** | нейтральные неоновые "шашки" (см. `images/`) | неоновая шайба | трек из Geometry Dash | карточка обведена **неоновым свечением** (акцент, что это дефолтный режим) |
| Пончик | **Sprinkle Slam** | пончики разных цветов | панкейк | Sunny — Boney M | по бокам карточки — по одному стикеру пончика (см. `stickers/`) |

Выбор — клик по карточке подсвечивает её (тем же механизмом, что уже реализован для 6 режимов в прототипе — нужно просто сократить список до 3 и сменить тексты/id).

### 3.4 Sound Overlay

Открывается по клику на Sound. Тонкий прямоугольник со скруглёнными краями = слайдер громкости. Слева от него — знак **минус**, справа — знак **плюс** (это новое требование, в текущем прототипе слайдер уже есть и работает мышкой, но знаков +/- по краям ещё нет — нужно дорисовать).

### 3.5 Play → выбор счёта (2 варианта, не 4)

По клику Play — сверху всплывают **две** карточки (не четыре, как в текущем прототипе):

- **Free Game** — очки просто считаются, игра не останавливается по счёту.
- **First To Five** — до 5 очков.

**Важно (см. раздел 5.4):** сервер сейчас **всегда** останавливает игру при счёте 5:5 независимо ни от чего (`max_score = 5` жёстко зашито). То есть "Free Game" физически не бесконечна на текущем бэкенде — это нужно решить с бэкенд-разработчиком (либо он делает `max_score` настраиваемым/бесконечным, либо "Free Game" на первом этапе просто тоже играется до 5, но без явного "победного" UI-финала). Отметь это Вике как вопрос к бэкендеру, не решай сама молча.

### 3.6 Play → выбор стика (2 варианта, не 4)

Снизу экрана всплывает панель, заголовок **"Choose Your Stick"** по центру сверху панели. Два варианта стика (конкретные картинки зависят от выбранного режима — см. таблицу 3.3 и папку `images/`). Выбранный увеличивается и подсвечивается — механизм такой же, как уже реализован в прототипе для 4 стиков, просто сузить до 2 и подключить реальные картинки вместо кружков-заглушек.

После выбора счёта и стика — переход на Экран 2.

## 4. Экран 2 — игровое поле

Общая механика (координаты, столкновения, голы) одна для всех режимов — задаётся сервером (раздел 5). Меняется только визуальное оформление (фон, цвет неона, спрайты стика/шайбы/ворот).

### 4.1 Разметка (одинаковая для всех режимов, судя по backend-константам)

- Поле: прямоугольник со скруглёнными углами, `LEFT_WALL=-200 .. RIGHT_WALL=200` по X, `DOWN_WALL=-300 .. TOP_WALL=300` по Y (координаты сервера, центр поля — `(0,0)`).
- Центральный круг + перекрестие в центре, средняя линия по горизонтали.
- Ворота — вырез в верхней и нижней стенке между `GOAL_LEFT=-100` и `GOAL_RIGHT=100`, обозначены скруглённой рамкой другого цвета (в мокапах — фиолетовая/сиреневая неоновая скобка).
- Счёт — две цифры друг под другом в неоновой рамке-боксе, справа от поля (см. мокапы `images/`).

### 4.2 Оформление по режимам (из реальных мокапов, см. `images/`)

- **Cosmic Rink**: фон — глубокий космос со звёздами, обводка поля — циановый неон, рамки ворот — фиолетовый/пурпурный неон, центральный круг — белый неон. Стик игрока 1 — Земля, стик игрока 2 — Марс, шайба — Луна.
- **Neon Velocity**: чёрный фон, обводка — циан, рамки ворот — пурпур, центр — циан с зелёным светящимся крестом (это и есть неоновая шайба в состоянии покоя). Стики — два концентрических неоновых круга разного цвета на игрока (синий / маджента). В мокапе есть жёлтый квадрат со знаком "?" (лутбокс) — **и ещё один жёлтый значок-ромб со стрелкой ">", назначение которого не описано в ТЗ — уточни у Вики, что это** (возможно артефакт мокапа, не отдельный игровой объект).
- **Sprinkle Slam**: фон — розово-персиковая глазурь с потёками шоколада по углам, обводка/центр — кремовый неон, рамки ворот — светло-сиреневый неон. Стик — пончик (розовая глазурь для игрока 1; **для игрока 2 нужно перекрасить глазурь в другой цвет — своей картинки под второго игрока в ассетах пока нет**, см. раздел 8). Шайба — панкейк с маслом и сиропом.

### 4.3 Игровые эффекты

- Борт при контакте с шайбой — вспышка/искра **своим цветом** (предположительно — акцентным неоновым цветом обводки поля данного режима).
- Ворота светятся **красным** при пропущенном голе и **зелёным** при забитом (с точки зрения того игрока, чей экран — т.е. это чисто визуальная реакция на клиенте на основе `GoalStatus`, сервер этот цвет не присылает явно).
- Лутбокс — стикер, появляется в случайном месте поля. Пойман стиком → рандомно увеличивает шайбу ИЛИ ворота. **На сервере лутбокса нет вообще** (ни спавна, ни подбора, ни рандома) — см. раздел 5.5, это целиком не реализованная на бэкенде фича.
- Победа/поражение — стикер `visual_effects/"You Won"` или `"You Lost"` (готовы, неоново-мультяшный стиль, см. папку).

### 4.4 Звуки (все готовы, см. `sound_effects/`)

Один и тот же звук для удара шайбы об борт ИЛИ об стик. Один и тот же звук гола вне зависимости от того, кто забил. Отдельный звук для сбора лутбокса. Отдельный звук победы (со стикером победы) и отдельный звук поражения (со стикером поражения).

## 5. Backend — реальный контракт (не черновой, как в v1 — это уже написанный код в `server/`)

### 5.1 Протокол

Один WebSocket-эндпоинт: **`ws://<host>/ws_connect`**. Никакого REST для старта игры, никакого выбора комнаты/режима на сервере — это просто один канал, первые два подключившихся клиента становятся `player1` и `player2` автоматически.

Клиент отправляет (JSON, судя по всему на каждый тик ввода):
```json
{ "position": { "x": 153.2, "y": 421.6 } }
```
Это позиция **своего стика** в игровых координатах сервера (см. 4.1 — те же `LEFT_WALL/RIGHT_WALL/TOP_WALL/DOWN_WALL`).

Сервер шлёт в ответ (после того как оба игрока подключились, ~120 раз в секунду, `time_delta = 1/120`):
```json
{
  "type": "GameState",
  "data": {
    "player1": {"position": {"first": 100, "second": 250}, "speed": 5, "speed_vector": {"first": 1.0, "second": 0.0}},
    "player2": {"position": {"first": -100, "second": -50}, "speed": 5, "speed_vector": {"first": 0.0, "second": 1.0}},
    "puck":    {"position": {"first": 100, "second": 250}, "speed": 5, "speed_vector": {"first": 1.0, "second": 0.0}},
    "score":   {"first": 2, "second": 3}
  }
}
```
Кроме `GameState` есть `type: "Message"` (текстовые уведомления — конец игры по счёту, конец игры по дисконнекту) и `type: "Error"`. Клиент должен уметь различать `type`, а не считать, что всегда придёт `GameState`.

### 5.2 Важно про "зеркалирование" — прочитать перед тем, как писать сетевой клиент

Сервер **сам разворачивает координаты и данные для player2**, чтобы оба клиента могли рендерить себя одинаково — как будто "я всегда внизу, я всегда player1". Конкретно:
- Входящие координаты от player2 сервер умножает на `-1` перед обработкой.
- Исходящий `GameState` для player2 — это тот же стейт, но `player1`/`player2` местами, все позиции/векторы скорости умножены на `-1`, и `score.first`/`score.second` тоже переставлены местами.

**Вывод для клиента:** не нужно самостоятельно определять, "я player1 или player2" и разворачивать что-либо на своей стороне — сервер уже прислал тебе всё в системе координат "я внизу, мой счёт — `score.first`". Просто рендери `data.player1` как себя, `data.player2` как соперника, `data.score.first` как свой счёт.

### 5.3 Игровые константы (уже зашиты в `server/constants.py`, дублировать их же и на клиенте для скейлинга рендера)

```
PUCK_RADIUS = 20        PLAYER_RADIUS = 50
LEFT_WALL = -200         RIGHT_WALL = 200
TOP_WALL = 300           DOWN_WALL = -300
GOAL_LEFT = -100         GOAL_RIGHT = 100
PLAYER_SPEED_LIMIT = 9   PUCK_SPEED_LIMIT = 6
```
Единицы — условные игровые, не пиксели. Клиенту нужен свой коэффициент масштабирования игровые-единицы → пиксели экрана.

### 5.4 Известные пробелы бэкенда — не решать самостоятельно, зафиксировать как вопросы к бэкенд-разработчику

1. **`max_score` жёстко = 5** (константа класса `GameMaster`, не параметр). Это значит: "First To Five" сервер уже поддерживает как есть. "Free Game" (бесконечный счёт) — **сервер сейчас не умеет**, игра всё равно закончится на 5:5. Нужно either сделать `max_score` настраиваемым по запросу клиента, either отдельный флаг "бесконечная игра".
2. **Нет выбора режима/скина на сервере.** Сервер ничего не знает про Cosmic Rink / Neon Velocity / Sprinkle Slam — это чисто визуальный выбор на клиенте, физика одна и та же для всех. Открытый вопрос: если игрок 1 выберет "Cosmic Rink", а игрок 2 — "Sprinkle Slam", у них будут просто разные скины одного и того же матча — это ок для air hockey (скин не влияет на физику), но стоит проговорить с командой, ожидаемо ли это, или скин должен синхронизироваться.
3. **Нет комнат/лобби.** Только один глобальный матч на весь сервер: третий подключившийся клиент получит отказ (`websocket.close()`). Если нужно несколько параллельных матчей — это отдельная фича бэкенда, сейчас её нет.
4. **Лутбоксов на сервере нет вообще** — ни спавна, ни позиции, ни события подбора. Это полностью не реализованная фича, а не "недоделанная" — с нуля для бэкенда.
5. **Нет reconnect** — обрыв соединения у любого игрока сразу завершает матч.
6. Комментарий в коде утверждает, что скорость игрока считается "за 1/60 секунды", но по факту сервер использует последние 2 полученных пакета без проверки временного интервала между ними — если клиент шлёт `position` реже/чаще 60 раз в секунду, расчёт скорости на сервере будет неточным. **Практический вывод для клиента: отправлять свою позицию стабильно ~60 раз в секунду**, не чаще и не реже, пока это не поправят на сервере.

### 5.5 Как запустить сервер локально для тестов

```
cd server
python3.13 -m venv .venv
.venv/bin/pip install -r requirements.txt
.venv/bin/uvicorn source:app --reload
```
Проверить, что жив: `curl http://127.0.0.1:8000/` → `{"message": "server root"}`.

## 6. Ассеты — краткий каталог (полное описание см. в CURSOR_PROMPT.md)

- `images/` — 5 файлов: мокапы полей (Cosmic Rink, Neon Velocity+лутбокс, Sprinkle Slam) + отдельные примеры стика/шайбы для Sprinkle Slam.
- `stickers/` — 4 боковых стикера для карточек режимов (космос ×2, пончик ×2). **Один файл (правый стикер космоса, Сатурн) — сток с водяным знаком pngtree, не для продакшена, см. раздел 8.**
- `visual_effects/` — 2 файла: "You Won" / "You Lost" попап-стикеры, готовы.
- `music/` — 3 mp3, фоновая музыка по одному треку на режим.
- `sound_effects/` — 5 mp3: удар, гол, лутбокс, победа, поражение.

## 7. Прогресс на сегодня

`prototype/aerohockey_interactive.py` уже кликабельный и уже содержит **больше, чем было в v1** — кто-то (Вика или уже Cursor) добавил overlay выбора счёта и панель выбора стика поверх старой версии. НО это сделано по старой спеке: 6 режимов вместо 3, 4 варианта счёта вместо 2, 4 généric-стика вместо 2 тематических, кнопки всё ещё тил вместо синих. Это не значит "переписать с нуля" — механизм подсветки/выбора уже рабочий и переиспользуемый, нужно только:
1. сократить `MODES` до 3 (Cosmic Rink / Neon Velocity / Sprinkle Slam),
2. сократить `SCORE_OPTIONS` до 2 (Free Game / First To Five),
3. сократить стики до 2 и подставить реальные картинки вместо кружков-заглушек,
4. перекрасить заливку кнопок в синий,
5. добавить тень на иконки, +/- по краям слайдера звука, неоновую обводку карточке Neon Velocity, боковые стикеры у Cosmic Rink/Sprinkle Slam.

## 8. Изменения с v1 (что переигралось)

- Цвет кнопок: тил → **синий** (подтверждено).
- Иконки: только play жёлтый → **все иконки жёлтые + тень**.
- Режимов: 6 → **3** (Cosmic Rink, Neon Velocity, Sprinkle Slam). Остальные три из v1 (Цветочный/Баскетбол/Бутик) в новом ТЗ не упоминаются вообще — если они ещё в планах, уточни у Вики, не придумывай сама.
- Вариантов счёта: 4 → **2** (Free Game, First To Five).
- Вариантов стика: 4 → **2**, теперь тематические (не абстрактные "Стик 1-4").
- Экран 2 теперь описан подробно и подкреплён реальными мокапами полей — в v1 его не было вообще.
- Появился реальный backend-код взамен чернового "контракта данных" из v1 (раздел 5).
- Зона ответственности Вики расширилась с "только Экран 1" до "весь клиент, включая Экран 2 и сетевой код".

## 9. Открытые вопросы (актуальные, свежие — не путать со снятыми из v1)

1. Точный HEX синего для кнопок/неона — не указан, нужен от Вики или дизайнера.
2. "Шесть полупрозрачных прямоугольников" в тексте ТЗ vs фактически 3 описанных режима — подтвердить, что это 3, не 6.
3. Судьба режимов из v1 (Цветочный/Баскетбол/Бутик) — в плане или нет.
4. Значение второго жёлтого значка (ромб со стрелкой ">") на мокапе Neon Velocity.
5. Сатурн-стикер с вотермаркой pngtree — заменить перед продакшеном.
6. Стикеры пончика разного стиля (фотореалистичный шоколадный дуэт vs плоский мультяшный розовый) — унифицировать стиль.
7. Нет ассета пончика для второго игрока (другой цвет глазури) — нужно сделать.
8. `max_score` жёстко=5 на сервере — конфликт с идеей "Free Game" (см. 5.4.1).
9. Синхронизация выбора режима/скина между игроками — нужна или нет (см. 5.4.2).
